# Autoresearch Concept and Analogues  
Karpathy’s **autoresearch** is an AI-driven loop that autonomously modifies code, runs experiments, and keeps only improvements【1†L279-L287】【43†L269-L278】.  For example, Karpathy’s agent repeatedly “reads the current code, makes a change (an experiment), runs it, measures the result (a hard number), and keeps the change if the score improved, else reverts”【43†L269-L278】.  Crucially, Karpathy uses a fixed **plan file** (`program.md`) containing human-written instructions (the “skill”) that the agent follows【45†L303-L306】.  In our case, step 1 (a frozen ablation-plan file) mirrors this: a static set of instructions/goals for the experiments, which the agent *will not modify* – analogous to Karpathy’s human-edited `program.md`【45†L303-L306】.   

Balukosuri’s “universal skill” project shows this pattern applied to code tasks beyond ML models: it scans a repo, defines metrics, and then enters an automated loop to improve something (here, a prompt)【43†L304-L312】【43†L321-L324】.  Together, these works demonstrate that giving an agent a fixed objective and letting it iteratively run experiments is feasible: you need (a) an objective metric to optimize, (b) an automated evaluation function, and (c) something to change (code or parameters)【43†L284-L293】. 

# Automating Kaggle Notebook Runs (CI)  
Each iteration’s core is running a Jupyter notebook on Kaggle and reacting to its status. The Kaggle CLI fully supports this workflow. For example, one can use `kaggle kernels update -p <path>` to upload and execute a notebook on Kaggle【47†L198-L206】.  The command `kaggle kernels status <user>/<kernel>` polls the run’s status, and `kaggle kernels output <user>/<kernel> -p <dir>` fetches all output files or logs once it finishes【47†L204-L214】.  In practice, one would script: upload/run the notebook, poll `kernels status` every minute, and if it *fails*, run `kernels output` to retrieve error logs. This pattern is proven: Nami Yousef’s GitHub Action for Kaggle integration tests does exactly that【57†L1-L4】. It uses `kaggle kernels status` to check success or failure and on error fetches logs with `kaggle kernels output`【57†L1-L4】.  

Several open-source tools exemplify this. The **KevKibe/kaggle-script-action** is a GitHub Action that automates running Python scripts (including entire notebooks) on Kaggle via CLI【37†L258-L266】. It supports configuring GPU/TPU, data sources, and even injecting custom code into a kernel; it then monitors the kernel status and retrieves logs【37†L268-L277】. Similarly, **namiyousef/action-kaggle-gpu-test** sends notebooks to Kaggle, polls status, and fetches outputs as shown in its docs【57†L1-L4】.  All indicate that automating Kaggle runs (steps 2–4) is *feasible* and common in CI/CD pipelines. The Kaggle CLI cheat sheet further confirms the commands needed:  
- **Upload & run:** `kaggle kernels update -p <path>`  
- **Check status:** `kaggle kernels status <user>/<kernel>` shows “running, complete, failed”【47†L210-L214】.  
- **Retrieve output/logs:** `kaggle kernels output <user>/<kernel> -p <dir>` downloads the run’s outputs【47†L204-L212】.  

# Tracking Experiments with Weights & Biases  
Tracking metrics (step 6) is fully supported: although Kaggle enforces *offline mode* (no internet during runs), one can log metrics locally and sync later. The open-source tool **kaggle-wandb-sync** does exactly this【22†L39-L47】. By setting `WANDB_MODE=offline` in the notebook, W&B records logs to files in the Kaggle output directory. After the run, a single CLI (`kaggle kernels output` to fetch files, then `wandb sync`) uploads all metrics to the W&B cloud【22†L39-L47】. This tool even integrates into GitHub Actions, automating “push → wait → output → sync” as one command【22†L39-L47】【22†L67-L74】. In short, tracking every notebook run in W&B is practical using these existing workflows.

# Documentation and Review (Audit)  
For step 7, generating docs and auditing, recent AI tools exist. For example, **automated notebook documentation** can be done with AI: the GitHub project *automatic-notebook-documentation* reads code and markdown cells and produces formatted documentation【54†L249-L257】. In other words, after each run you could feed the `.ipynb` into such a tool to produce a human-friendly summary. For auditing the results/changes, AI code-review agents have been built. Kurkowski describes an LLM-based code audit system that applies multiple “perspectives” (e.g. architectural, quality, security) to automatically review a repo and generate a report【34†L81-L90】. Analogously, one could use an LLM prompt to critique the new notebook/results (e.g. “Does this model meet our metrics? Any bugs or deviations from the plan?”). In practice, these steps (documentation and auditing) are feasible using existing AI assistants: one can auto-generate docs from notebooks【54†L249-L257】 and use LLM-based code analysis to review changes【34†L81-L90】.

# Feasibility and Existing Projects  
All pieces are technically feasible and several projects already cover parts of this pipeline:  

- **Kaggle CI/CD:** The KevKibe *kaggle-script-action* (GitHub) automates pushing notebooks to Kaggle, running them with GPU/TPU, polling status, and fetching logs【37†L258-L266】. Similarly, *namiyousef/action-kaggle-gpu-test* demonstrates running tests on Kaggle via `kaggle kernels update`, checking `kaggle kernels status`, and using `kaggle kernels output` to get logs on failure【57†L1-L4】. These confirm you can embed Kaggle runs in automated workflows.  
- **W&B Integration:** The `kaggle-wandb-sync` tool (GitHub/PyPI) is exactly designed for Kaggle+W&B experiment management【22†L39-L47】. It handles offline logging and syncing. This project shows clearly that logging and tracking metrics is solved.  
- **Karpathy’s Autoresearch:** While this focuses on ML code, it is the conceptual prototype for autonomous research loops【1†L279-L287】【43†L269-L278】. Your project is an extension of this idea to notebook-based Kaggle experiments.  
- **Universal Autoresearch Skill:** Balukosuri’s “Autoresearch Universal Skill” adapts Karpathy’s idea to arbitrary codebases (optimizing prompts across repos)【43†L304-L312】【43†L321-L324】. It shows the generality of the loop (generate, evaluate, keep or revert).  
- **Notebook Reliability:** Mario Caesar’s blog demonstrates best practices for production-ready Kaggle notebooks – e.g. idempotency, logging, error handling – to make the pipeline trustworthy【52†L1-L4】【23†L75-L82】. This confirms that robust Kaggle workflows are attainable.  
- **Auto-Documentation and Audit:** The “Automatic Documentation for Jupyter Notebooks” project (GitHub) shows one can auto-generate docs from notebooks【54†L249-L257】. LLM-based code auditors (e.g. Kurkowski’s agent) show automated review is possible【34†L81-L90】.  

In summary, all parts of **“AR by HV”** are practicable: Kaggle’s CLI and community tools support CI-style notebook runs【47†L204-L214】【57†L1-L4】, W&B tracking is solved by offline logging tools【22†L39-L47】, and AI assistants can help automate documentation and auditing【54†L249-L257】【34†L81-L90】. Several open-source projects (linked above) implement overlapping ideas, but no single project integrates *all* these features end-to-end. Your proposed pipeline would combine them into a coherent autoresearch system for notebooks. Each element – automated runs on Kaggle, iterative code updates, metric tracking, and report generation – has precedent in existing tools and research【1†L279-L287】【37†L258-L266】【22†L39-L47】【54†L249-L257】.
